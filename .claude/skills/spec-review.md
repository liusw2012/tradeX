## 评估维度

### 1. 需求完整性 (Requirements Completeness)
- 功能需求覆盖：核心功能是否完整描述
- 非功能需求：性能、安全、可靠性要求
- 用户故事：是否有清晰的用户场景
- 边界条件：异常流程是否覆盖

### 2. 需求可行性 (Feasibility)
- 技术可行性：现有技术栈是否支持
- 资源可行性：人力、时间是否充足
- 依赖项：外部依赖是否明确

### 3. 需求可测试性 (Testability)
- 验收标准：是否有明确的验收条件
- 测试用例：是否可转化为测试用例
- 量化指标：性能指标是否可测量

### 4. 需求一致性 (Consistency)
- 术语统一：专业术语是否一致
- 逻辑自洽：需求之间是否无矛盾
- 优先级：需求优先级是否清晰

## 量化指标

| 指标 | 计算方式 | 通过阈值 |
|------|----------|----------|
| 需求覆盖率 | (已覆盖需求数 / 总需求数) × 100% | >= 90% |
| 验收标准明确率 | (有明确验收条件的需求数 / 总需求数) × 100% | >= 85% |
| 场景覆盖率 | (已覆盖用户场景数 / 总场景数) × 100% | >= 80% |
| 风险项数量 | 列出所有识别的风险 | <= 5 |

## 评估标准

- **通过**: 需求覆盖率 >= 90% 且验收标准明确率 >= 85%
- **警告**: 需求覆盖率 >= 80% 且验收标准明确率 >= 75%
- **不通过**: 需求覆盖率 < 80% 或验收标准明确率 < 75%

```
- 总体得分: XX%

## 详细评估

### 1. 需求完整性
- 得分: XX%
- [详细分析]

### 2. 需求可行性
- 得分: XX%
- [技术风险列表]

### 3. 需求可测试性
- 得分: XX%
- [可测试性分析]

### 4. 需求一致性
- 得分: XX%
- [一致性问题]

## 风险汇总
- [风险列表及影响]

## 评审结论
[通过/警告/不通过] - 详细说明
```

## 执行步骤

1. 读取SPEC文档内容
2. 识别并列出所有功能需求
3. 检查每个需求的验收标准
4. 识别用户场景和边界条件
5. 评估技术可行性和风险
6. 生成评审报告

## 注意事项

- 重点关注核心业务需求
- 识别模糊或歧义性描述
- 检查需求的相互依赖关系
- 使用中文输出报告
