## 评估维度

本评估基于 **ISO 25010** 软件质量模型、**Google C++ Style Guide**、**CERT C++ Secure Coding**、**MISRA C++** 等业界标准，涵盖以下十大维度：

---

### 1. 编程规范 (Coding Standards)

#### 1.1 命名规范
- 变量命名：snake_case / camelCase 一致性
- 函数命名：动词前缀，清晰表达行为
- 类/结构体命名：PascalCase，名词表达实体
- 常量/枚举：kCamelCase 或 ALL_CAPS
- 命名空间：全小写，层次清晰

#### 1.2 代码格式
- 缩进：4空格或Tab一致性
- 行长度：不超过100-120字符
- 大括号：K&R风格一致性
- 空格：运算符周围空格规范化
- 行尾：统一LF或CRLF

#### 1.3 头文件管理
- #include 顺序：标准库 → 第三方 → 项目内部
- 头文件保护：#pragma once 或 #ifndef 保护
- 前置声明：优先使用前置声明减少编译依赖
- 最小化包含：只包含必要的头文件

#### 1.4 现代C++特性
- 智能指针：unique_ptr / shared_ptr 正确使用
- auto关键字：适当使用避免冗余类型
- range-for：优先使用范围for循环
- constexpr：编译期常量使用
- std::optional / std::variant：空值安全处理

#### 1.5 const正确性
- 成员函数：方法是否标记const
- 参数：输入参数是否const引用
- 局部变量：是否尽可能const

### 2. 安全性检测 (Security)

#### 2.1 内存安全 (对应CWE-119系列)
- 缓冲区溢出：strcpy/strcat/sprintf 不安全函数
- 越界访问：数组下标边界检查
- 释放后使用：检查空指针/野指针
- 双重释放：资源释放路径检查
- 内存泄漏：new/delete 配对检查

#### 2.2 输入验证 (对应OWASP)
- SQL注入：参数化查询检查
- 命令注入：系统调用输入验证
- 路径遍历：文件路径规范化检查
- 整数溢出：算术运算溢出检查
- 格式化字符串：printf系列安全使用

#### 2.3 敏感信息保护
- 密钥硬编码：密码/密钥/Token检查
- 日志敏感信息：敏感数据脱敏
- 内存敏感信息：安全擦除
- 配置泄露：配置文件敏感信息

#### 2.4 资源管理 (RAII)
- 异常安全：异常发生时资源正确释放
- 锁管理：std::lock_guard / std::unique_lock
- 文件资源：RAII封装
- 网络资源：连接正确关闭
- 线程资源：线程正确join/detach

#### 2.5 并发安全 (对应CWE-362系列)
- 数据竞争：共享变量访问保护
- 死锁：锁顺序一致性检查
- 竞态条件：原子操作使用检查
- 线程安全：线程局部存储正确使用

---

### 3. 性能评估 (Performance)

#### 3.1 不必要的拷贝
- 拷贝构造函数：深拷贝必要性检查
- 参数传递：值传递 vs 引用传递
- 返回值：移动语义使用
- 临时对象：RVO/NRVO优化
- 容器拷贝：emplace vs insert

#### 3.2 循环优化
- 循环内调用： invariant code motion
- 预分配：reserve / resize 预分配
- 迭代器：前置++ vs 后置++
- 范围遍历：std::for_each vs 手写循环

#### 3.3 锁与并发优化
- 锁粒度：粗粒度 vs 细粒度
- 无锁编程：原子操作使用
- 读写锁：std::shared_mutex
- 条件变量：避免忙等

#### 3.4 内存优化
- 内存池：高频分配场景内存池
- 对象池：复用频繁创建对象
- 缓存：LRU缓存使用
- 内存对齐：访问效率优化

#### 3.5 算法复杂度
- 时间复杂度：O(n) vs O(n²)
- 空间复杂度：额外空间使用
- 数据结构选择：容器选择合理性

#### 4.1 错误处理

## 问题分级标准

### Critical - 致命问题 (必须修复)
- 内存泄漏导致资源耗尽
- 数据竞争导致未定义行为
- 安全漏洞导致数据泄露
- 死锁导致系统完全不可用
- 缓冲区溢出导致可利用漏洞

### Blocker - 阻塞问题 (必须修复)
- 违反核心安全编码规范
- 明显的内存安全问题
- 资源未正确释放(RAII违规)
- 严重的性能问题(O(n²)未优化)
- 错误处理严重缺失

### Warning - 警告问题 (应该修复)
- 代码风格不一致
- 可读性较差
- 复杂度较高(>15)
- 注释不充分
- 轻微的性能问题

### Info - 建议问题 (可选修复)
- 代码格式细节
- 轻微的命名问题
- 建议性的重构
- 文档完善建议

## 通过条件

代码审查通过的**必要条件**：
1. **整体通过率 >= 85%**
2. **安全性通过率 >= 90%**
3. **无Critical级别问题**
4. **无Blocker级别问题**
5. **Warning级别问题数量 <= 30**
6. **所有10个维度均 >= 70%**

## 执行步骤

1. **扫描代码**：扫描项目中所有源代码文件(.cpp, .h)
2. **静态分析**：使用工具进行自动化分析
   - 编译检查：clang-tidy
   - 格式检查：clang-format
   - 安全检查：cppcheck
3. **人工审查**：对关键代码进行人工审查
4. **问题分类**：将发现的问题分类到十大维度
5. **量化评分**：计算各项指标得分
6. **生成报告**：生成详细评估报告

|------|------|--------|

| clang-format | 格式检查 | 代码格式/缩进 |
| cppcheck | 静态分析 | 内存安全/空指针 |
| Coverity | 商业静态分析 | 安全/质量 |

## 注意事项

- 使用Grep工具搜索代码模式来发现问题

- 评估应覆盖所有核心模块
- 生成报告时使用中文
- 重点关注Critical和Blocker问题
- 结合项目实际情况调整阈值
